### Live updates without a hand-written SignalR Hub

PRD FR-16/NFR-9 ask for a hand-off on one station's screen to appear on another station's screen, on a different PC, without a manual refresh. This does **not** need a custom SignalR Hub: Blazor Server already holds one persistent SignalR circuit per browser tab. `IRelayNotifier` (singleton, `Application/Relay`) is a plain C# `event Action<int> StageChanged`; `RelayService` raises it with the affected stage's ID after a successful claim or hand-off. `StationRun.razor` subscribes in `OnInitializedAsync`, and on a matching event calls `InvokeAsync(() => { refresh; StateHasChanged(); })` тАФ `InvokeAsync` is required because the event fires from whatever circuit's context triggered the change, not the subscribing component's own circuit. Unsubscribe in `DisposeAsync`, or you leak a delegate reference to a dead component on a singleton that outlives every circuit. Verified working with two real, separately-attached Playwright browser tabs (no shared navigation): advancing Station 1 through a hand-off updates Station 2's already-open tab with no `page.goto`/reload involved.

### A SQLite WAL gotcha when resetting the local dev database

If you delete `App_Data/flowline.db` to force a clean reseed, also delete `flowline.db-shm` and `flowline.db-wal` (WAL-mode sidecar files). Deleting only the main file and leaving a stale `-wal` behind causes SQLite to replay the *old* WAL into the new file on next open, silently reintroducing old seed data (hit this once тАФ looked like new seed code wasn't running, when actually it ran fine but old data came back from the leftover WAL). `rm -f App_Data/flowline.db*` gets all three.

### Running on SQL Server (multi-provider)

The app runs on SQLite (dev default) or SQL Server (company deployment), chosen by the `DatabaseProvider` config key + the `FlowLine` connection string тАФ no code edit. `Program.cs` switches on it: `Sqlite` тЖТ `UseSqlite`, `SqlServer` тЖТ `UseSqlServer(...).MigrationsAssembly("FlowLine.Migrations.SqlServer")`. EF Core migrations are **provider-specific and can't share an assembly** (one model snapshot per DbContext per assembly), so SQLite migrations live in `FlowLine.Infrastructure/Data/Migrations` and SQL Server migrations live in the separate `FlowLine.Migrations.SqlServer` project. On startup `Database.Migrate()` creates all seven FlowLine tables + seeds automatically on *either* provider тАФ verified against SQL Server 2022 in Docker. To run on SQL Server locally: `docker run -e ACCEPT_EULA=Y -e MSSQL_SA_PASSWORD='FlowLine!Dev2026' -p 1433:1433 -d mcr.microsoft.com/mssql/server:2022-latest`, then set `DatabaseProvider=SqlServer` and `ConnectionStrings__FlowLine="Server=localhost,1433;Database=FlowLine;User Id=sa;Password=FlowLine!Dev2026;TrustServerCertificate=True"` as env vars. When adding a migration, add it to **both** providers (each with its own `--project`, see Commands) or the two snapshots drift. The `Guid` `RowVersion` concurrency token works unchanged on SQL Server (don't switch it to a native `rowversion`; the app-managed pattern is provider-agnostic).

### The two company-owned tables (`History`, `StaffTable`)

These pre-exist in the company SQL Server (verified live against the real `xumlocal` database) and are **not owned by FlowLine**. Mapped in `ConfigureExternalTables` (`FlowLineDbContext.OnModelCreating`) with `ToTable(..., t => t.ExcludeFromMigrations())` so FlowLine's migrations *never* emit DDL against them тАФ `Database.Migrate()` stays responsible only for FlowLine's own tables (the seven core ones plus `WorkflowAssignments`). `StaffTable` maps cleanly: `StaffNumber` (int key, `ValueGeneratedNever`), `Name`, `TestingPower`. **`History`'s real schema is messier than a clean orders queue** (it's the company's ~123k-row product-testing log), so its mapping does real work: the C# property names differ from the columns (`OrderId`тЖТ`Orderid`, `Status`тЖТ`TestStatus`, `AssigneeNumber`тЖТ`AssignedNumber`), and several columns are fixed-width `nchar`/`varchar` holding space-padded or non-numeric values. Value converters in `ConfigureExternalTables` handle this on read: a `trim`/`trimNullable` converter strips the `nchar` padding (`Orderid`, `Channel`, `TestStatus`, `TestedBy`, `SKU`, and `StaffTable.Name`); `ncharToInt` parses the `nchar` `QTY` to `int` (blank/non-numeric тЖТ 0); and `stringToNullableInt` parses the `varchar` `AssignedNumber` to `int?` (its values are usually non-numeric markers like `'Unknown'`, so **assignee-name resolution is usually null** тАФ don't expect it to populate). All converters are identity in the to-provider direction because these tables are read-only. Usage is **read-only for `History`** (order source for import *and* the prebuild lookup, matched on `Orderid`); `StaffTable` drives login + name lookup and тАФ per user request, the one sanctioned write-back тАФ is written by `StaffService.CreateStaffAsync`/`UpdateStaffAsync` (manager staff management, `/admin/staff`), touching only the three mapped columns and never deleting rows. Under SQLite these tables are absent (`ExcludeFromMigrations` means even `EnsureCreated` skips them), which would break login/import/prebuild in dev тАФ so `SeedData.SeedDevExternalTablesAsync` creates and seeds mock copies via raw `CREATE TABLE IF NOT EXISTS` **only** on SQLite, with columns/types mirroring the real schema (`Orderid`/`TestStatus`/`AssignedNumber`, `QTY`/`AssignedNumber` as text); tests do the same via `SqliteTestDatabase.CreateExternalTablesAsync`, so the SQLite tests exercise the exact same converters. **Known follow-up:** `OrderImportService.GetImportableOrdersAsync` does `db.History.OrderByDescending(...).ToListAsync()` тАФ fine against the 5-row mock, but it materializes all ~123k rows on the real server; it needs a server-side filter/cap (criteria TBD with the company) before the import page is usable in production. Any *further* write-back (e.g. stamping `IsTested`/`TestStatus` on `History`) is a deliberate new decision тАФ the design was explicitly chosen as read-only to avoid touching production company data, and the `StaffTable` staff-management writes are the single user-sanctioned exception so far.

### Auth, roles, and the login gotcha

Cookie auth, staff-code only (no password тАФ PRD ┬з4 permitted a stubbed auth; the company asked for this richer version). **The gotcha:** a live Blazor Server SignalR circuit can't call `HttpContext.SignInAsync` (the response is already flushed), so login is a *static SSR* form (`Login.razor`) that POSTs to the `POST /auth/login` minimal-API endpoint in `Program.cs`, which validates the code via `IStaffService`, writes claims (`NameIdentifier`=staff number, `Name`, `flowline:level`), and redirects. Both `/auth/*` endpoints are `AllowAnonymous().DisableAntiforgery()`. Authorization is enforced by `AuthorizeRouteView` (in `Routes.razor`) against each page's `[Authorize]`: **`[AllowAnonymous]` does NOT override `[Authorize]` in `AuthorizeRouteView`**, so `Login.razor` is exempted by *location* (it lives in `Components/`, outside the `Pages/` folder whose `_Imports.razor` applies `[Authorize]`), not by attribute. `Routes.razor`'s `NotAuthorized` distinguishes not-signed-in (тЖТ `RedirectToLogin`, a full-load nav to `/login`) from signed-in-but-wrong-level (тЖТ a "No access" message, so an operator isn't bounced to the login form). **That branch only catches in-app navigations** тАФ a *direct URL load* of a gated page is rejected by endpoint authorization before Blazor's router runs, and lands on the cookie handler's `AccessDeniedPath` instead; that's `/denied` (`Components/Denied.razor`, anonymous-by-location like `Login.razor`), which shows the same "No access" message. It used to be `/login`, which confusingly re-showed the sign-in form to an already-signed-in operator. Access matrix тАФ L1: run assigned workflows only; L2 (`CanViewReports`): + read-only workflow list + Timing; L3 (`CanManage`): everything. Nav items and edit controls are gated with `<AuthorizeView Policy="...">`; pages carry matching `[Authorize(Policy=...)]`; URL-level guards in `Home`/`WorkflowStations`/`StationRun` stop an L1 reaching a non-assigned workflow by typing the URL. Read claims off the principal with `ClaimsPrincipalExtensions.GetLevel()`/`GetStaffNumber()`.

### Entry-stage modes: prebuild scan and free-run

Two mutually-exclusive `Workflow` flags change how the **first stage** gets work (a normal workflow just auto-claims from the queue). Both are toggled in the builder (`WorkflowEdit.razor`), enforced mutually-exclusive server-side in `WorkflowBuilderService` (turning one on clears the other), copied by `DuplicateWorkflowAsync`, and honoured by `StationRun.RefreshAsync` (an entry station in either mode does **not** auto-claim тАФ work starts only from the scan/button):

- **`RequiresPrebuild`** тАФ the entry station shows a scan box; a unit is started by the scan. See "Prebuild scanning" below.
- **`AllowAdHocStart`** (per user request тАФ free-run/training mode; migration `AddWorkflowAllowAdHocStart` in both providers) тАФ for routines/training where the point is *doing and timing the steps*, with no premade order. The entry station shows a "Start run" button (`ShowAdHocStart`; Enter also triggers it); clicking it calls `IRelayService.StartAdHocAsync(stationId)`, which creates a WorkItem with a generated `OrderNumber` (`RUN-yyyyMMdd-HHmmss`), SKU `"RUN"`, qty 1, already `InProgress`/claimed at the station, so the operator begins immediately and every step is timed like any other unit. Throws `RelayOperationException` if the station isn't the workflow's first stage or the workflow isn't in ad-hoc mode.

### Prebuild scanning

A `Workflow.RequiresPrebuild` flag (toggled in the builder) changes how that workflow's **first stage** gets work: instead of pulling from a queue, its station shows a scan box and a *unit is started by the scan*. `StationRun`'s `ShowPrebuildScan` is true when the station is the workflow's entry stage (lowest `Stage.OrderIndex`), the workflow requires a prebuild, and nothing's active; the box is auto-focused so a barcode scanner's keystrokes+Enter land there. On scan, `IRelayService.CreateFromPrebuildAsync(stationId, prebuildId)` resolves the ID against **two sources, in priority order** (per user request тАФ orders authored on the Orders screen must be scannable too): (1) a WorkItem in this workflow with that `OrderNumber` that's already Queued *at this entry stage* (Orders-screen creation or History import) is simply **claimed** to the scanning station тАФ no duplicate created, its authored SKU/qty kept; (2) otherwise the ID is looked up in `History` (matched on `OrderId`) and a WorkItem is created inheriting that row's `SKU`/`QTY`/`Channel` (with `OrderNumber` = the scanned ID), already `InProgress`/claimed so the operator starts immediately. Either way it then hands off downstream like any other unit. It throws `RelayOperationException` if the ID matches neither source, the station isn't the workflow's first stage, or a matching non-Completed WorkItem is anywhere past the entry queue (claimed or mid-line тАФ a duplicate scan). The WorkItems-first order matters: an order imported from History into the queue exists in *both* sources, and claiming the queued one is what prevents a duplicate. For a prebuild workflow, the entry station deliberately does **not** auto-claim queued units тАФ work starts only on a scan. The scanned ID is assumed to match a `History.OrderId`; if the company keys prebuilds off a different column, it's a one-line change. (There is intentionally no `WorkItem.PrebuildId` column тАФ the scanned ID *is* the WorkItem's `OrderNumber`; an earlier design that stored it separately was removed in migration `DropWorkItemPrebuildId`.)

**Mock-table drift on old dev DBs:** the SQLite mock `History`/`StaffTable` aren't in migrations, so `CREATE TABLE IF NOT EXISTS` in `SeedDevExternalTablesAsync` silently keeps whatever shape an old dev DB already had тАФ hit this: a DB seeded before the real-schema mapping still had `History."Status"`/`"Assigne Number"`, and every History query crashed with "no such column". The seeder now detects a `History` table missing any of `Orderid`/`TestStatus`/`AssignedNumber` (via `pragma_table_info`) and drops/recreates the mock. If the mock schema changes again, extend that column list.

### The stats dashboards (`/admin/stats`, Chart.js)

Three-level drill-down, all `[Authorize(Policy = CanViewReports)]`: workflow picker тЖТ workflow dashboard тЖТ per-operator page. Mechanics worth knowing before touching them:

- **Charts are Chart.js 4, vendored locally** (`wwwroot/lib/chartjs/chart.umd.min.js` тАФ LAN app, no CDN at runtime) and driven through `wwwroot/js/flowline-charts.js` (`window.flowlineCharts.durationBar/durationGroupedBar/countLine/countBar`, values in seconds). The wrapper keeps a per-canvas registry and **destroys the previous Chart instance before re-rendering** тАФ Chart.js throws on canvas reuse, and Blazor re-invokes the render on every date-range change. Pages call it from `OnAfterRenderAsync` guarded by a `_chartsPending` flag set in `OnParametersSetAsync` (never from `OnInitializedAsync` тАФ JS interop isn't available during prerender).
- **The date filter is URL state, not component state**: `from`/`to` query params (yyyy-MM-dd, local days), read via `[SupplyParameterFromQuery]`, converted to a UTC instant range by `StatsDateRange` (`FlowLine.Web`). `StatsDateControls.razor` (shared, in `Components/`) applies presets/custom ranges by navigating to the same page with updated params тАФ so data reloads happen in `OnParametersSetAsync` (a same-page query change doesn't re-run `OnInitializedAsync`), drill-down links append `StatsDateRange.QuerySuffix(...)` to keep the range, and browser back/forward just works.
- Formatting lives in `StatsFormatting` (`FlowLine.Web`): durations as "1h 03m"/"4m 05s", deltas as "12% faster/slower" with `delta-faster`/`delta-slower` CSS classes (green/red).
- Legacy data renders honestly: completions recorded before `StartedAtUtc`/`CompletedByStaffNumber` existed show "тАФ" durations and appear in no staff list, but still count in units-built and step totals.

### Concurrency token note

SQLite has no native rowversion/timestamp type, so `WorkItem.RowVersion` (`Guid`) is an *application-managed* concurrency token: `FlowLineDbContext.SaveChanges`/`SaveChangesAsync` bump it to a new `Guid` for every modified `IConcurrencyAware` entity before calling `base.SaveChanges`, and it's marked `.IsConcurrencyToken()` in `OnModelCreating`. This is the standard EF Core pattern for providers without native rowversion support тАФ don't replace it with a `[Timestamp]`/`IsRowVersion()` attribute, that only works on SQL Server. `RelayService.ClaimNextAsync` catches `DbUpdateConcurrencyException` and retries against the next-oldest queued WorkItem (PRD ┬з6.5); `AdvanceAsync` deliberately does *not* retry on conflict тАФ a concurrent advance on the same already-claimed WorkItem indicates a genuine double-call (e.g. a debounce failure), and the exception should propagate so the caller re-reads current state rather than silently double-processing a step.

### DateTime, not DateTimeOffset, for all timestamps

`WorkItem.QueuedAtUtc`/`CreatedAtUtc` and `StepCompletion.CompletedAtUtc` are `DateTime` (always UTC by convention), not `DateTimeOffset`. This isn't a style choice: the SQLite EF Core provider throws `NotSupportedException` translating `ORDER BY` over a `DateTimeOffset` column, and `ClaimNextAsync`'s "oldest queued" query and `TimingService`'s ordering both need to order by timestamp. Hit this once already (see migration `ChangeTimestampsToDateTime`) тАФ don't reintroduce `DateTimeOffset` on a column you intend to filter/sort by.

### Why timing needs `CreatedAtUtc`, not just `QueuedAtUtc`

`WorkItem.QueuedAtUtc` is overwritten on every hand-off (it means "queued at the *current* stage," for `ClaimNextAsync`'s FIFO ordering тАФ see the entity's XML doc). By the time a WorkItem reaches Completed, all of its earlier per-stage queued-at values are gone; only the most recent one survives. That makes it useless for reconstructing stage 1's start time after the fact, which is why `CreatedAtUtc` exists as a separate, *never-updated* field set once at creation. `TimingService` computes stage N's dwell time as `(last StepCompletion in stage N) тИТ (last StepCompletion in stage N-1)`, using `CreatedAtUtc` as the stand-in end-of-"stage 0" for stage 1 тАФ this way stage 1 isn't a special case, and the per-stage durations sum exactly to `CompletedAtUtc тИТ CreatedAtUtc`. If you add a new WorkItem-creation path anywhere (besides `OrderService.CreateWorkItemAsync` and `SeedData`), don't forget `CreatedAtUtc` тАФ it defaults to `DateTime.UtcNow` on the entity, so it's only a problem if you ever explicitly overwrite it after construction.

### `@bind` on a `Dictionary<TKey,TValue>` indexer needs the key pre-seeded

`@bind="someDictionary[key]"` is valid (the indexer has both a getter and setter, so it's an assignable target) but Blazor's generated code calls the getter on every render тАФ if the key isn't present yet, that's an unhandled `KeyNotFoundException` that takes down the whole page (`WorkflowList.razor`'s per-row "duplicate as" field and `WorkflowEdit.razor`'s per-stage "new step name" field both do this). Fix is to seed every key with `TryAdd(key, string.Empty)` when the parent collection loads, before any indexer-bound input renders. Hit this in both places building the workflow builder UI тАФ if you add another per-row/per-item text input bound to a dictionary, seed it the same way.

### `@inject` can't reuse the component's own class name

A Razor component's generated class is named after its file (`Orders.razor` тЖТ class `Orders`). `@inject IOrderService Orders` inside that file fails to compile with `CS0542: 'Orders': member names cannot be the same as their enclosing type` тАФ the injected property collides with the class itself. Name the property something else (`OrderService` worked fine) rather than mirroring the service's domain name; this only bites when the component file and the injected interface happen to share a name; it didn't apply to `IRelayService`/`IWorkflowBuilderService` since none of the page filenames matched.

### EF Core in-memory tracking can mask the FK guard it means to test

When testing that a delete is correctly *blocked* by a Restrict FK (e.g. deleting a Stage that still has a WorkItem on it), don't seed the blocking row through the *same* `FlowLineDbContext` instance the method under test will use. EF Core auto-fixes-up navigation properties between any two tracked entities sharing a FK, regardless of which `Include` path loaded them тАФ so the blocking row ends up attached to the in-memory object graph, and removing the parent throws `InvalidOperationException: ... has been severed ...` from EF's own change tracker *before* the call ever reaches the database. That's not the same failure as the real one (`DbUpdateException` from SQLite's FK constraint), and `WorkflowBuilderService`'s catch block won't catch it, so the test fails even though production code is correct тАФ a fresh request/circuit in production always gets its own DbContext with no such stale tracking. Use a second, short-lived `FlowLineDbContext` to seed the conflicting row (see `WorkflowBuilderServiceTests`'s `DeleteStageAsync`/`DeleteStepAsync` guard tests), mirroring how two separate requests would actually behave.

## What FlowLine is

A self-hosted LAN web app that runs guided, multi-station assembly/teardown lines (replacing a legacy Google Apps Script + Google Sheet tool). The key design upgrade: **process definitions are data, not code** тАФ an admin defines workflows (stages, steps, instructions, images) entirely through the UI (`/admin/workflows`), with no redeployment.

## Target tech stack (per PRD ┬з9)

- **ASP.NET Core 8 + Blazor Server** (C#) тАФ UI and logic live together server-side; each browser keeps a live SignalR connection. This is *not* a Blazor WebAssembly or separate JS-frontend app тАФ that tradeoff was explicitly considered and rejected.
- **EF Core** as the data access layer тАФ must abstract the DB provider so SQLite can later be swapped for the company database via config/connection-string change, not a rewrite (NFR-2, NFR-3).
- **SQLite (WAL mode)** for local dev тАФ single file on the server. **SQL Server** is also supported for the company deployment via the config-driven provider (see "Running on SQL Server" above); the NFR-2/NFR-3 "swap the provider, not a rewrite" goal is realised.
- **Local filesystem** folder for uploaded media (`MediaAsset` rows store paths, not blobs).

## Deployment model тАФ read this before designing anything network-related

FlowLine is **one server process** on a LAN machine; every station and the admin UI are just browser clients pointing at it (e.g. `http://flowline:5000`). There is no per-station install and no file syncing between machines. A change at one station (e.g. a hand-off) must propagate to other stations' screens in real time via SignalR, not polling. Keep this client-server topology in mind тАФ don't design features that assume a single local user/browser.

## Domain model (PRD ┬з6) тАФ the core abstraction to preserve

Two sides, deliberately separated:

**Template side** (authored by admin, defines a process):
```
Workflow 1тФАтФАтФА* Stage 1тФАтФАтФА* Step 1тФАтФАтФА* MediaAsset
Workflow 1тФАтФАтФА* Stage 1тФАтФАтФА* Station   (a station mans one stage)
```
- `Workflow` тАФ a named process (e.g. "Gaming PC Build").
- `Stage` тАФ an ordered station-position within a workflow (old "Step 1тАУ4").
- `Step` тАФ an ordered sub-task within a stage, with instructions (old "1.1 Pick Case"). `RequiresScan` (toggled per step in the builder; migration `AddStepRequiresScan` in both provider assemblies) makes completing it demand a barcode scan matching the WorkItem's OrderNumber тАФ enforced in `RelayService.AdvanceAsync` (its `scannedCode` param), not just the UI; the station screen swaps the Advance button for an auto-focused scan box on such steps.
- `MediaAsset` тАФ image/GIF attached to a `Step`.

**Instance side** (runtime, the orders flowing through):
```
Workflow 1тФАтФАтФА* WorkItem (CurrentStage, Status, ClaimedByStation)
WorkItem 1тФАтФАтФА* StepCompletion *тФАтФАтФА1 Step
```
- `Station` тАФ a physical screen bound to one `Stage` of one `Workflow`.
- `WorkItem` тАФ a unit in flight (Order #, SKU, Qty, Channel), tracking `CurrentStageId`, `Status` (Queued / In Progress / Completed, plus terminal Cancelled тАФ admin void тАФ and Scrapped тАФ failed on the line; both free the order number for re-import/re-scan and are excluded from stats), `ClaimedByStationId`, and a row-version concurrency token.
- `StepCompletion` тАФ one row per completed sub-step per WorkItem: step, station, `CompletedByStaffNumber` (the signed-in operator тАФ a plain int, not an FK, since `StaffTable` is external; passed from `StationRun` into `AdvanceAsync`), `StartedAtUtc` (when the step actually began: the previous completion at the same stage, or `WorkItem.ClaimedAtUtc` for a stage's first step тАФ so `CompletedAtUtc тИТ StartedAtUtc` is the step's true duration, excluding queue wait), and `CompletedAtUtc`. `WorkItem.ClaimedAtUtc` (set on claim/prebuild scan, cleared on hand-off/completion) exists solely to anchor that first step. Both new columns are nullable тАФ null means "recorded before this was tracked". This is the source of truth for cycle-time and who-did-what data тАФ never go back to "timestamp in a positional column."

There is **no separate queue table**: a stage's queue is just "WorkItems with `CurrentStageId` = this stage and `Status` = Queued."

## The relay model тАФ the central correctness requirement (NFR-4)

This is the most important behavior to get right; read PRD ┬з6.5 before touching claim/hand-off logic. Implemented in `RelayService` (`src/FlowLine.Application/Relay`):

1. **Claim** (`ClaimNextAsync`): a free station claims the oldest Queued WorkItem at its stage (ordered by `QueuedAtUtc`, not `Id` тАФ see the entities' XML docs) via an atomic, guarded conditional update (`WHERE Status = Queued AND ClaimedByStationId IS NULL`) plus the optimistic concurrency token. Exactly one station wins if two race; the loser's `SaveChanges` throws `DbUpdateConcurrencyException`, which the method catches and retries against the next-oldest candidate. Two stations on the same stage can never claim the same unit.
2. **Work** (`AdvanceAsync`): determines the WorkItem's next outstanding step itself (from existing `StepCompletions` at the current stage) rather than trusting a caller-supplied step ID, and writes a `StepCompletion` with a server timestamp.
3. **Hand-off:** completing a stage's last step is **one transaction** that sets `CurrentStageId` to the next stage, resets `Status` to Queued, refreshes `QueuedAtUtc`, and clears the claim тАФ or marks the WorkItem Completed if there's no next stage.
4. **Buffering:** stages naturally buffer WorkItems when downstream is slower; nothing is lost or overwritten.
5. **Live updates:** when a WorkItem lands in a stage's queue, that stage's station screen(s) update with no manual refresh, via `IRelayNotifier` (see "Live updates without a hand-written SignalR Hub" below) тАФ verified working across two separate browser tabs.

A claim or hand-off must never be a non-atomic read-then-write тАФ that's exactly the bug class (row deletes, no locking) the legacy system had and FlowLine is meant to fix.

## Scope discipline

The PRD explicitly excludes (┬з4) for the prototype: company DB/ERP integration, real auth/roles (admin/worker distinction may be stubbed), multi-site/cloud/HA, native mobile apps, and analytics dashboards beyond raw timing + a basic view. Don't build toward these unless asked тАФ the milestone sequence (┬з10) targets M4 (a second, structurally different workflow running with no code changes) as the definition of prototype success. **Note:** the company-DB integration (SQL Server provider, `History`/`StaffTable`) and real auth/roles (staff-code login, three access levels, workflow assignment) have since been built *at explicit user request*, deliberately going beyond these ┬з4 exclusions тАФ they're no longer out of scope, but everything else in ┬з4 still is.

Several questions are explicitly open and **assumed resolved one way** for the prototype unless told otherwise (┬з11): workflows are linear (no branching), and one station per stage is the default (multi-station-per-stage is allowed by the data model but not necessarily exposed in the UI). The "work only moves forward" assumption has since been **relaxed at user request**: stations can send a unit back to an earlier stage for rework (`RelayService.SendBackAsync`) and scrap failed units тАФ see the Relay bullet.
